# 商用差距报告（当前仓库）

## 1. 结论摘要
- 当前项目具备商用雏形，但尚未达到“可审计可追溯”的上线门槛。
- 主要风险在于：组件层演示硬编码、来源元数据缺失、规则版本不可追踪。

## 2. 模块级差距

### 2.1 八字（bazi）
- 已有：四柱、十神、藏干、纳音、空亡、神煞、大运。
- 差距：
  - 结果页仍混有静态示例数据。
  - 缺字段级 sourceRef。
  - 缺 ruleSetVersion 与 ruleId。

### 2.2 六壬（luren）
- 已有：本地规则库与断语库。
- 差距：
  - 断语缺条文级 citation（卷/篇/条）。
  - 缺标准化接口层，页面直接消费本地常量。

### 2.3 经方本草（herbal）
- 已有：source/original 字段，古籍出处较完整。
- 差距：
  - 部分辨证映射仍为组件内硬编码。
  - 缺统一 sourceRef 结构。

### 2.4 中医首页（tcm）
- 已有：功能入口和声明。
- 差距：
  - AI结果为组件内固定文本，无法商用审计。
  - 未完全接入经典数据结构。

## 3. 商用整改优先级

### P0（必须先做）
- 建立统一数据契约（已提供 docs/COMMERCIAL_DATA_SCHEMA.md）。
- 将组件内演示数据迁移至数据层。
- 所有结论补充 sourceRef + ruleId + ruleSetVersion。

### P1（建议做）
- 建立 API adapter 层（前端仅消费 schema）。
- 增加质量校验：缺字段/空数据/来源缺失即阻断渲染。

### P2（增强项）
- A/B 规则版本灰度。
- 指标看板：结论命中率、回溯一致性。

## 4. 你需要提供的最小输入
- 竞品清单：产品名 + 官网 + 对标功能。
- 后端能力：API 地址或 mock 网关。
- 规则版本策略：如 bazi-rules@1.0.0。
- 法务边界：可引用古籍范围与标注要求。
